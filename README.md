# Дипломный проект. Автоматизация тестирования API сервиса Stellar Burgers

Данный репозиторий содержит автотесты для проверки ключевого бизнес-функционала API космической бургерной **Stellar Burgers**. Работа выполнена в рамках финального проекта курсов автоматизации тестирования на Java в Яндекс Практикуме.

---

## 🛠️ Стек технологий

* **Java 11** — основной язык разработки.
* **JUnit 4** — фреймворк для написания и запуска тест-кейсов.
* **RestAssured** — библиотека для автоматизации тестирования REST API.
* **Allure Reports** — инструмент для генерации наглядных и детальных отчётов о тестировании (с использованием аннотаций `@Step`, `@DisplayName`, `@Description`).
* **Java Faker** — библиотека для генерации реалистичных и уникальных тестовых данных (Email, Имя, Пароль).
* **Lombok** — плагин для автоматической генерации POJO-методов (геттеры, сеттеры, конструкторы).
* **Maven** — сборщик проекта и менеджер зависимостей.

---

## 🗺️ Покрытие тестами

В соответствии с техническим заданием автоматизированы три ключевых бизнес-блока:

### 1. Создание пользователя (`CreateUserTest`)
* Успешное создание уникального пользователя (с валидацией вложенных полей ответа).
* Получение ошибки `403 Forbidden` при попытке зарегистрировать уже существующего пользователя.
* Получение ошибки `403 Forbidden` при попытке создания пользователя с незаполненным (или `null`) обязательным полем (`email`, `password`, `name`).

### 2. Логин пользователя (`LoginUserTest`)
* Успешный вход в систему под существующей учётной записью (проверка выдачи `accessToken` и `refreshToken`).
* Получение ошибки `401 Unauthorized` при попытке входа с неверным паролем.
* Получение ошибки `401 Unauthorized` при попытке входа с неверным email.

### 3. Создание заказа (`CreateOrderTest`)
* Успешное создание заказа авторизованным пользователем (с глубокой валидацией привязки заказа к `owner`).
* Успешное создание заказа неавторизованным пользователем (проверка короткого формата JSON ответа).
* Успешное создание заказа с передачей корректного списка ингредиентов.
* Получение ошибки `400 Bad Request`, если заказ отправлен без ингредиентов (пустой список).
* Получение ошибки `500 Internal Server Error`, если в запросе передан невалидный/сломанный хэш ингредиента.

## 🏗️ Архитектура проекта

Проект построен по принципам чистого кода и разделения зон ответственности:
* `praktikum.model` — POJO-модели для сериализации и десериализации JSON-тел (`UserModel`, `UserLogin`, `OrderModel`).
* `praktikum.data` — класс конфигурации и динамической генерации данных `UserData` на базе `Java Faker`.
* `praktikum.steps` — изолированные шаги для выполнения HTTP-запросов (`UserSteps`, `OrderSteps`), размеченные Allure-аннотациями.
* `praktikum.BaseApiTest` — базовый класс, управляющий глобальным `baseURI` и обеспечивающий автоматическую очистку данных (`@After` удаление пользователя по Bearer токену после каждого теста) согласно принципу DRY.
